2026 web app stack: choose boring cores, modern edges

Bottom line: use a lean TypeScript product stack, not a microservice checklist. Keep Postgres central, split services only where it buys isolation or scaling, and choose tools that reduce operational surface area without trapping the whole product in one platform.

Recommended default

SvelteKit or TanStack Start · Hono · Postgres + Drizzle · Neon/Supabase · Better Auth or Clerk · Trigger.dev/Inngest · Cloudflare + Railway · Sentry + PostHog + OTel

Do not default to

Next.js + Vercel + Prisma + Kubernetes just because it is common. It can be right, but it is not the simplest or most performance/DX-weighted default in 2026.

Deployment shape

Start with 3–4 deployables: web, API/BFF, durable workers, and observability/release pipeline. Add microservices after boundaries are proven.

What changed after research

A generic answer would rank popularity and produce a familiar stack. Direct checks of docs, pricing, runtime constraints, and negative evidence shifted the answer toward: fewer services, more durable workflows, explicit cost controls, and reversible platform choices. The user asked for microservices, caches, queues, auth, CI/CD, IaC, logging, observability, and deployment — the researched answer is that most of those should be covered, but not all by separate services on day one.

Important decisions and tradeoffs

Frontend framework

SvelteKit is the cleanest answer when React is not required: less component ceremony, integrated routing/rendering, Vite HMR, and strong docs. TanStack Start is the React bet with the best architecture fit for typed routing and client-first apps, but the RC/newer status means you should spike it before committing. Next.js is not “outdated”; it is just not the simplest default if you do not explicitly want React Server Components and Vercel’s platform.

Microservices

Do not build “microservices” as a checklist. Build modular boundaries: web, API/BFF, workers, and shared packages. Split into separately owned services only for different scaling profiles, security/compliance isolation, or teams. Most early microservice complexity shows up as distributed auth, logging, migrations, retries, and deployment choreography.

Database and state

Keep important state in Postgres. Use Redis/Upstash for cache, rate limits, ephemeral queues, and locks — not as the source of truth. Use platform stores like Cloudflare KV/Durable Objects for edge-specific patterns, not portable core domain state.

Deployment economics

Cloudflare Workers pricing is compelling for web/edge request workloads: paid plan minimum $5/month, 10M included requests, and no Workers egress charge in the checked docs. Railway is easier for normal containers and background workers but charges by CPU/RAM/egress and gates log retention by plan. Fly gives global VM control, but with more infrastructure responsibility. Vercel is excellent for Next-first teams, but pricing complexity and platform coupling should be a deliberate choice.

Negative evidence and caveats

  • React Server Components complexity: powerful, but cache/data-fetching semantics can tax DX.
  • Edge limits: great latency and cost for web requests; poor fit for long CPU, native dependencies, raw sockets, and some DB access patterns.
  • Auth liability: self-owned auth is not free — security reviews, password/session policy, MFA, account recovery and audit trails matter.
  • Observability cost: replay, traces, logs, and events can grow faster than app traffic. Set sampling and retention early.
  • New-stack churn: TanStack Start, Better Auth, Trigger.dev-style workflows and serverless databases are promising, but pin versions and avoid unnecessary lock-in.

Implementation roadmap

  1. Week 0 spike: build one auth-protected vertical slice: login, CRUD, background email/task, analytics event, error capture, preview deploy.
  2. Month 1: add migrations, seed data, CI checks, Sentry/PostHog dashboards, spend limits, backup/restore test.
  3. Before production: load-test DB connection behavior, queue retries/idempotency, auth recovery flows, webhook replay, and deployment rollback.
  4. After traction: split workers/services only where measurements prove need; introduce IaC for durable shared resources.

Methodology

Scope included frontend, backend, storage, auth, background work, cache, observability, deployment, CI/CD, IaC, and supporting APIs. I used official docs/pricing first, GitHub/activity checks where network access allowed, community/adversarial search as a risk-finding pass, and excluded choices mainly justified by popularity or premature enterprise complexity.
