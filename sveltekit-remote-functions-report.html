What “remote functions” means in current SvelteKit

Official docs describe remote functions as a tool for type-safe communication between client and server. They are exported from .remote.js or .remote.ts files and can be called anywhere in the app, but always execute on the server. On the client, SvelteKit transforms imports into generated fetch wrappers.

They are currently experimental, available since SvelteKit 2.27, and require opt-in via kit.experimental.remoteFunctions plus Svelte’s experimental async support. That matters: a repo can be impressive and recent, but remote-function APIs may still shift.

query

Read dynamic server data; request/client dedupe; refreshable; has batching for N+1 avoidance.

form

Server mutations via progressively enhanced forms, Standard Schema validation, field issues, and query refresh behavior.

command

JS-only mutations callable from handlers, useful where no-JS fallback is not required.

prerender

Build-time/static data variant with cache behavior and optional dynamic fallback.

Public discussion / forum sentiment

Overall: sentiment is enthusiastic but cautious. Users like the reduction in boilerplate and the native tRPC-like feeling. Repeated praise: easier code navigation, less API-route ceremony, remote forms feel especially valuable, and the feature feels like something SvelteKit “needed”.

Main caveats from discussions: experimental instability, changing cache semantics, testing difficulties, auth guard footguns, no built-in streaming, and confusion around async Svelte integration. Production advice is split: some users say they are using it successfully, others recommend waiting until it is stable for serious apps.

  • SvelteKit PR #13986: implementation/design discussion; merged into SvelteKit 2.27 era.
  • PR #15779 — isomorphic caching for queries: evidence cache semantics are still being shaped.
